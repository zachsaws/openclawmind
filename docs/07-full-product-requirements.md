# Bluff Arena 全站开发需求文档

## 1. 项目目标

开发一款可上线的 Web 棋牌游戏产品，主题是“人类 vs 多模型 AI 的诈唬牌对战”。

目标不是做 MVP 演示，而是直接做一个可用、可部署、可扩展、可继续上线运营的完整产品。

这个产品的核心卖点：

- 玩家可以和 3 个 AI 模型同桌打牌。
- AI 不只是出牌，还会在聊天区进行心理博弈。
- 整体交互体验要尽量贴近主流斗地主 / 掼蛋产品的用户习惯。
- UI 必须是游戏化牌桌，不接受后台管理页、表单页、调试页风格。

## 2. 目标用户

- 热爱 AI 行业的玩家。
- 对大模型人格、策略差异、模型间博弈感兴趣的人。
- 熟悉斗地主 / 掼蛋 / 跑得快这类棋牌游戏交互方式的普通玩家。

## 3. 产品定位

一句话定义：

这是一个让人类玩家与 3 个不同人格或不同厂商大模型同桌进行诈唬牌对战的实时 Web 棋牌游戏，玩家通过出牌、质疑、聊天和心理博弈来争取胜利。

## 4. 成功标准

产品完成标准不是“页面能打开”，而是以下链路全部跑通：

- 用户能进入首页并快速开局。
- 用户能与 3 个 AI 完成一整局对战。
- 对战过程包含出牌、质疑、聊天、倒计时、结算、再来一局。
- 页面是游戏牌桌式体验，不是原型工具界面。
- 后端支持实时同步、基础风控、日志、健康检查、部署。
- 项目具备继续接数据库、模型 API、登录和云端部署的结构基础。

## 5. 核心玩法规则

### 5.1 基本规则

- 固定 4 人对局：1 人类 + 3 AI。
- 使用 2 副扑克牌，含大小王，共 108 张。
- 4 人均分，每人 27 张。
- 随机先手。
- 当前轮到的“报点”按顺序推进。
- 玩家出牌时可以选择 1 到 4 张牌。
- 玩家报出的点数必须匹配当前轮到的报点。
- 出牌后进入 `challenge_window`，其他玩家可以质疑。
- 若质疑成功，虚报者吃掉桌上牌堆。
- 若质疑失败，质疑者吃掉桌上牌堆。
- 某玩家手牌出完后，若挑战窗口结束且未被翻掉，则该玩家获胜。

### 5.2 聊天规则

- 每回合允许发送聊天消息。
- 聊天用于心理战，不影响规则判定。
- 聊天内容需要经过审核。
- 禁止违规、违法、辱骂、仇恨、色情、诈骗等内容。

### 5.3 可见性规则

- 玩家只能看到自己的手牌明细。
- AI 的手牌只能看到剩余张数，不能看到牌面。
- 牌桌中央显示当前牌堆数量和最近一手信息。

## 6. 交互与体验要求

### 6.1 总体要求

- 必须模仿主流斗地主 / 掼蛋产品的操作习惯，而不是做通用网页应用交互。
- 用户第一眼要感觉是“游戏”，不是“系统”。
- 交互要求低学习成本，尽量不需要阅读大量规则说明。

### 6.2 界面风格要求

- 首页 / 牌桌 / 结果页采用强游戏化视觉。
- 牌桌要有明确的桌面中心区、四个座位区、底部手牌区、固定动作条。
- 手牌必须可点击选择。
- 出牌、质疑、不质疑、提示、再来一局等按钮位置稳定。
- 聊天区和对局日志分开展示。
- 要有倒计时进度条。
- 牌堆应在牌桌中间可见，最好显示背面叠牌。

### 6.3 必须具备的交互习惯

- 一键“快速开局”。
- 底部固定手牌区。
- 手动选牌后点击“出牌”。
- 有“提示”按钮帮助选牌。
- 有“不质疑”快捷操作。
- 有结算弹窗和再来一局。
- 有新手引导或首次提示层。
- 有明显的“轮到谁”视觉提示。

### 6.4 动效要求

- 页面加载时有适度入场动效。
- 出牌时要有明显反馈，最好有牌飞向桌面中心的感觉。
- 倒计时和轮转要有过渡反馈。
- 结算时要有胜负状态反馈。

## 7. AI 能力要求

### 7.1 AI 参与方式

- 3 个 AI 同桌对战。
- 每个 AI 可配置模型来源和人格。
- 人格至少支持：激进、稳健、嘴硬、沉默。

### 7.2 AI 行为

- AI 能进行出牌。
- AI 能进行质疑。
- AI 能发送聊天内容。
- AI 输出必须遵守合法动作范围，不能输出无效结构。

### 7.3 多模型适配要求

- 要有统一的 `LLMAdapter` 抽象层。
- 目标接入：OpenAI、Anthropic、Google。
- 支持超时、重试、降级。
- 模型不可看到不该看到的隐藏状态。

## 8. 前端需求

### 8.1 技术要求

- 使用现代前端技术栈实现。
- 默认 Web 产品，桌面和移动端浏览器均可用。
- 代码结构要可维护，不能把所有逻辑塞进单文件。

### 8.2 页面要求

- 首页 / 开局页
- 主牌桌页
- 结果页 / 结算弹窗
- 新手引导层
- 断线 / 重连 / 加载 / 错误状态

### 8.3 状态同步

- 优先使用 WebSocket 实时同步。
- 不依赖纯轮询作为主同步方式。
- 客户端需支持掉线重连和状态恢复。

## 9. 后端需求

### 9.1 基本服务能力

- HTTP API
- WebSocket 实时事件
- 游戏状态机
- AI 编排层
- 聊天审核
- 日志
- 健康检查

### 9.2 数据层要求

- PostgreSQL：持久化房间、玩家、对局、动作、事件、用户信息。
- Redis：房间在线状态、短期事件、会话、缓存、限流。
- 后续支持重连恢复。

### 9.3 安全与风控

- 基础登录体系，至少支持游客身份。
- 接口鉴权。
- 速率限制。
- 输入校验。
- 内容审核。
- 基础反刷和防滥用。

### 9.4 可观测性

- 结构化日志。
- 核心指标。
- 错误告警。
- 健康检查接口。

## 10. 接口与事件要求

当前冻结契约如下，若要调整必须成体系调整，不允许随意漂移：

### 10.1 HTTP API

- `POST /games`
- `POST /games/:id/start`
- `POST /games/:id/actions/play`
- `POST /games/:id/actions/challenge`
- `POST /games/:id/actions/chat`
- `POST /games/:id/rematch`
- `GET /games/:id/state`
- `GET /games/:id/replay`
- `GET /healthz`
- `GET /meta/bootstrap`

### 10.2 WebSocket 事件

- 客户端 -> 服务端：`game:subscribe`
- 服务端 -> 客户端：`game:update`

## 11. 工程要求

- 项目必须可以本地一键启动。
- 必须提供 Dockerfile。
- 必须提供环境变量模板。
- 必须有清晰 README。
- 必须有基础测试。
- 必须可以构建通过。
- 不接受“只能在某个人电脑上跑”的交付。

## 12. 测试要求

至少覆盖以下内容：

- 核心规则单元测试。
- 公共视图不泄露 AI 手牌。
- 2 副牌 4 人发牌正确。
- 开局到结算主流程测试。
- 再来一局流程测试。
- WebSocket 基本连通性测试。

- 不要做成管理后台风格。

- 不要只做前端不做后端。
- 不要只做原型不做实时同步。
- 不要返回“这里先留空，后续补”式关键链路。
- 不要出现违规、违法、低俗、辱骂类文案和内容。
